Legacy Rendszerek & Ügyfél Stratégia

Mit kezd az ügyfél a meglévő rendszereivel? És mi a 1337 szerepe?

A Helyzet

A magyar FSI ügyfelek (KAVOSZ, Dorsum ügyfelei, bankok) rengeteg legacy rendszert üzemeltetnek: régi saját fejlesztések, licenszelt szoftverek, SaaS megoldások.

Amikor AI platformot választanak, nem csak a platform számít — hanem az is, hogy mi lesz a meglévő rendszereikkel és hogyan illeszkedik a workflow-jukba.

A compliance kritikus: akár cloud, akár on-prem — MNB, GDPR, AI Act, DORA. Minden integrációnak audit-ready-nek kell lennie.

01 MCP mint Integrációs Réteg

Kérdés: Az MCP (Model Context Protocol) a megfelelő technológiai megközelítés a legacy rendszerekhez való kapcsolódásra?

Válasz: Igen, az MCP ideális erre a célra

Az MCP egy absztrakciós réteg, ami lehetővé teszi, hogy az AI egységesen kommunikáljon bármilyen háttérrendszerrel, függetlenül annak technológiájától.

  • Protokoll-agnosztikus: REST, SOAP, SQL, file-based — bármit be tud csomagolni
  • Tool-alapú: Egyszerű eszközöket definiálsz (list, get, update), az AI használja
  • Inkrementális: Rendszerenként külön MCP server, fokozatosan építhető
  • Audit-ready: Minden tool hívás loggolható, nyomon követhető
  • Izolált: Ha egy rendszer változik, csak az MCP server-t kell módosítani

Kulcs Insight

Az MCP lényege: a legacy rendszer bonyolultsága el van rejtve. Az AI csak annyit lát: "van egy tool ami lekéri az ügyféladatot". Nem tudja, és nem is kell tudnia, hogy az Oracle, COBOL, vagy REST API mögött van.

02 Ügyfél Stratégiai Opciók

Miután az MCP-n keresztül kapcsolódunk a legacy rendszerekhez, az ügyfélnek döntenie kell: mi legyen hosszú távon ezekkel a rendszerekkel?

A
Legacy Rendszer Újraírása
Greenfield fejlesztés, modern stack

Az ügyfél úgy dönt, hogy a régi rendszereket lecseréli modern, AI-native megoldásokra. Nem csak integrálódik az AI-hoz, hanem az AI köré épít új rendszert.

Előnyök

  • Technikai adósság megszűnik
  • AI-native architektúra
  • Jobb user experience
  • Hosszú távú költségcsökkenés
  • Modern compliance beépítve

Kockázatok

  • Magas upfront költség (€500k - €5M)
  • Hosszú timeline (12-36 hónap)
  • Business disruption risk
  • Szükséges szakértelem ritka
  • "Big bang" projektesély: 70% fail

A1: Ki fejleszti és üzemelteti?

In-house fejlesztés

  • Az ügyfél "szoftvercéggé válik"
  • Dev team felvétele (drága, nehéz)
  • Teljes kontroll, de teljes felelősség
  • KAVOSZ-típus: valószínűtlen választás

Outsourced (pl. 1337)

  • 1337 mint "AI-native szoftverfejlesztő"
  • Managed service modell
  • Ügyfél nem válik tech céggé
  • 1337 scope bővülés: platform → software

1337 Szerepe Opció A esetén

Ha az ügyfél újraír: 1337 lehet a fejlesztő partner és/vagy managed service provider. Ez jelentős scope bővülés — nem csak "AI platform", hanem "AI-native szoftver fejlesztés és üzemeltetés".

Kérdés: Akarunk ilyen irányba menni? Ez más üzleti modell.

B
Legacy Megtartása + AI Overlay
MCP integráció, workflow automatizálás

Az ügyfél megtartja a legacy rendszereket, de az AI platformon keresztül dolgozik velük. A régi UI-t egyre kevésbé használja, a workflow az AI felületen történik.

Előnyök

  • Nincs disruption, azonnal indul
  • Alacsony kockázat
  • Inkrementális értékteremtés
  • Legacy licensz marad, de kevesebb használat
  • ROI gyors (hetek, nem évek)

Kockázatok

  • Technikai adósság marad
  • Legacy korlátoz (amit nem tud, nem tud)
  • Dupla költség (legacy + platform)
  • Integráció bonyolultsága

1337 Szerepe Opció B esetén

Ez a core value proposition. 1337 építi az MCP szervereket, a domain-specifikus skill-eket, és deploy-olja a platformot.

A legacy rendszer "adatforrás és akció-végpont" lesz — az AI orkesztrál, a legacy végrehajtja.

C
Custom UI a Legacy Fölött
AI-powered, de nem agentic interface

Nem mindenki tud vagy akar "agentic" (chat-alapú) felületet használni. Egyes ügyfelek hagyományos UI-t szeretnének, de AI-támogatással.

  • Ismerős felület a usernek
  • Könnyebb adoption
  • Workflow-specifikus optimalizálás
  • Kevesebb "prompt engineering" a usertől
  • Guided experience

Kockázatok

  • Több fejlesztési munka (UI építés)
  • Kevésbé flexibilis
  • Minden workflow-hoz külön UI
  • Nem skálázódik annyi use case-re

Kulcs Insight: A Backend Ugyanaz

Opció B és C ugyanazt a backend-et használja: Skills + MCP Serverek. A különbség csak a frontend: chat vs. dashboard/forms. Egy ügyfélnek lehet MINDKETTŐ — power user-ek chat-elnek, back-office dashboard-ot kap.

1337 Szerepe Opció C esetén

1337 építi a custom UI-t is a platform fölé. Ez product development munka, nem csak platform deployment.

Alternatíva: Az ügyfél (vagy Dorsum-típusú partner) építi a UI-t, mi API-t adunk.

Stratégiai Javaslat

A legtöbb magyar FSI ügyfél a B opciót fogja választani (Legacy + AI Overlay), mert ez a legkisebb kockázat és leggyorsabb érték.

De fel kell készülni a C opcióra is (Custom UI), mert nem mindenki fog chat-alapú felületet használni — különösen back-office, ahol structured workflow a norma.

Short-term (0-6 hó)

Opció B: MCP + Skills + Agentic interface. KAVOSZ és Dorsum reference-ek.

Mid-term (6-18 hó)

Opció C képesség: Custom UI / Dashboard builder. Embedded AI widgets ügyfeleknek.

Long-term (18+ hó)

Döntés az A opcióról: Belépünk-e a "AI-native software development" piacra?

04 Nyitott Stratégiai Kérdések

Megválaszolandó

  • Építünk-e Custom UI / Dashboard képességet?
  • Vállalunk-e full software dev projekteket (Opció A)?
  • Mi a boundary: platform vs. application?
  • Partner modell: Dorsum épít UI-t, mi adjuk az AI-t?

Eldöntött

  • ✓ MCP a helyes integrációs megközelítés
  • ✓ Skills-based architecture
  • ✓ On-prem opció kell (LLM provider abstraction)
  • ✓ Compliance-first minden szinten
